# v0.1.6 Design — daily restart cap + terminal-state alert

- **Status:** Approved (2026-04-25)
- **Closes:** [#12](https://github.com/BananaBay69/claude-code-watchdog/issues/12), [#13](https://github.com/BananaBay69/claude-code-watchdog/issues/13)
- **Target release:** v0.1.6

## Context

The 2026-04-24 → 2026-04-25 OAuth recovery incident on the Mac mini dogfood target exposed two design gaps in the Phase 1 supervisor:

1. **No restart cap.** When watchdog couldn't recover the session via restart (because the underlying issue — OAuth credential corruption — was beyond restart's reach), it kept restarting every cooldown interval for ~15 hours. Cooldown paces but doesn't bound; ~30+ kill-session cycles plausibly contributed to credential damage via token-rotation race conditions.

2. **No detection for terminal / unrecoverable states.** When TUI displays `Not logged in` / `--channels ignored` / `Channels require claude.ai authentication`, restart cannot help (every fresh session reads the same broken keychain). v0.1.5's idle-defer logic correctly stops restarting but logs `OK: Session alive`, which is misleading: the bot is fully dead and only `/login` fixes it.

## Scope

Bundle into a single v0.1.6 release. Both issues share the same alert-emission infrastructure (`WATCHDOG_ALERT_CMD` hook); splitting into two PRs would force rebase conflicts on the alert plumbing.

**In scope:**

- `WATCHDOG_DAILY_RESTART_CAP` env (default 10) — cap daily restarts.
- `WATCHDOG_THROTTLED_COOLDOWN` env (default 3600) — extended cooldown after cap.
- `WATCHDOG_ALERT_CMD` env (default unset) — pluggable alert hook.
- Two-tier pattern detection: `RESTART_PATTERNS` (existing 7) + `TERMINAL_PATTERNS` (new 3).
- New CLI flags: `--reset`, `--status`.
- Alert dedup: edge-triggered (alert once per state-entry, suppress until clears).

**Out of scope:**

- Phase 2 plugin heartbeat extension.
- Centralized config schema (deferred per CONTRIBUTING.md, tracked separately as a future issue).
- Linux support (macOS-only stays).
- Re-alert / digest on persistent state (no trigger yet; could be added later if users miss the one-shot alert).

## Design decisions

| Decision | Choice | Rationale |
|---|---|---|
| Bundle vs split | **Bundle (one PR / one release)** | Both share alert hook; splitting causes rebase conflicts. Single CHANGELOG entry, single deployment. |
| After-cap behavior | **Throttled cooldown (1h)** | Hard halt is safer but loses transient-issue self-heal. Throttle balances damage-limit against recovery capability. |
| Default cap value | **10** | Healthy bots: 0–3 restarts/day. 10 is well above normal noise, well below pathological (incident hit 30+ in 12h). |
| Default throttle | **3600s (1h)** | Reduces post-cap rate by 12× (300 → 3600). Caps post-cap budget at ~22 attempts/day vs 144 unbounded. |
| Default cap on/off | **On (cap=10)** | Incident proves default-passive is unsafe. Setting `WATCHDOG_DAILY_RESTART_CAP=0` provides escape hatch for legacy unbounded behavior. |
| Alert mechanism | **Pluggable hook + log-only fallback** | Upstream stays generic. Downstream installs ship their own helper script (e.g. macOS Notification + Telegram, ntfy, email). |
| Refactor approach | **Modular** (extract helpers, add `--status`, add tests) | Code is becoming state-heavy. Inline approach hits readability cliff. |
| Testing | **Unit (source-mode) + Integration (subprocess + mock tmux) + CI** | Comprehensive coverage; CI prevents regressions. |
| Spec location | `docs/superpowers/specs/` in this repo | Lives with code; PR-portable. |

## Architecture

### Per-tick flow

```
launchd fires every 180s → claude-watchdog.sh main()
│
├── 1. Setup (config, log rotation) — unchanged
│
├── 2. Single pane capture: PANE=$(tmux capture-pane -p -S -50)
│
├── 3. Two parallel detections (orthogonal):
│     ├── RESTART_MATCH = detect_restart_pattern(PANE)    [yes:<pat> | no:]
│     └── TERMINAL_MATCH = detect_terminal_state(PANE)    [yes:<pat> | no:]
│
├── 4. Terminal-state branch (no restart, alert only):
│     ├── if TERMINAL_MATCH yes:
│     │     ├── if not alert_already_sent(not-logged-in):
│     │     │     emit_alert(not-logged-in, ...) + mark_alert_sent
│     │     └── else: silent (dedup)
│     └── if TERMINAL_MATCH no:
│           clear_alert_flag(not-logged-in)        [state recovered]
│
├── 5. Restart decision (existing case-stale/fresh logic, unchanged):
│     SHOULD_RESTART = 0 or 1
│
├── 6. Cap-aware restart (when SHOULD_RESTART=1):
│     ├── COUNT = read_restart_count()
│     ├── EFF_COOLDOWN = effective_cooldown(COUNT)
│     ├── if check_cooldown(EFF_COOLDOWN):
│     │     ├── start_claude()                     [bumps counter internally]
│     │     ├── if NEW_COUNT == CAP and not alert_already_sent(cap-reached):
│     │     │     emit_alert(cap-reached, ...) + mark_alert_sent
│     └── else: blocked by cooldown
│
├── 7. Case C (claude process dead) — same cap-aware path as 6
│
└── 8. log "OK: Session alive"
```

Two key invariants:

- Terminal-state alert is independent of restart logic. Same tick can both alert and restart.
- Cap modulates cooldown duration but doesn't replace it. `check_cooldown` still runs.

### State files

All under `$LOG_DIR/`, alongside existing `.watchdog-last-restart`.

| File | Format | Lifecycle |
|---|---|---|
| `.watchdog-restart-count-YYYYMMDD` | single-line integer | bumped atomically (temp + mv) on each `start_claude`; new file at local midnight; old files retained (small, no GC) |
| `.watchdog-alert-sent-cap-YYYYMMDD` | empty file (existence flag) | created on first cap-reached alert of the day; cleared by day rollover |
| `.watchdog-alert-sent-not-logged-in` | empty file (existence flag) | created on entry to terminal state; deleted on recovery (pattern no longer matches) |

**Two dedup regimes**:

- `cap-reached` is **time-based** (rollover at midnight). One alert per day max.
- `not-logged-in` is **state-based** (cleared when symptom disappears). Each state-entry triggers exactly one alert; recovery resets so future re-entry alerts again.

## Components

### Function inventory

**New:**

| Function | Purpose |
|---|---|
| `today_yyyymmdd()` | echo `date +%Y%m%d` |
| `read_restart_count()` | cat counter file, default 0 if missing or unreadable |
| `bump_restart_count()` | atomic +1 (temp file + mv) |
| `effective_cooldown(count)` | echo `COOLDOWN_SECONDS` if count < CAP, else `THROTTLED_COOLDOWN`. If `WATCHDOG_DAILY_RESTART_CAP=0`, always echo normal cooldown. |
| `alert_already_sent(key)` | return 0 if flag file exists, 1 otherwise |
| `mark_alert_sent(key)` | touch flag file |
| `clear_alert_flag(key)` | rm -f flag (no error if missing) |
| `emit_alert(type, msg)` | log `ALERT [$type]: $msg`. If `WATCHDOG_ALERT_CMD` set, invoke with `WATCHDOG_ALERT_TYPE` and `WATCHDOG_ALERT_MSG` env vars; non-zero exit → log `WARN`. Never blocks watchdog on alert failure. |
| `detect_restart_pattern(pane)` | echo `yes:<matched>` or `no:` (extract from existing inline grep) |
| `detect_terminal_state(pane)` | echo `yes:<matched>` or `no:` (new patterns) |

**Refactored:**

| Function | Change |
|---|---|
| `check_cooldown()` | accepts `$1` = effective cooldown; default 300 preserves backward-compat for any in-script callers |
| `start_claude()` | internally calls `bump_restart_count` after kill+new-session |
| Main flow | wrapped in `main()` function with source-guard at end of file (enables source-mode unit tests) |

### Pattern lists

```bash
RESTART_PATTERNS=(
    "rate-limit-options"
    "Enter to confirm.*Esc to cancel"
    "Yes, I trust this folder"
    "You've hit your limit"
    "resets [0-9]+[ap]m"
    "Press Enter to continue"
    "Do you trust the files"
)

TERMINAL_PATTERNS=(
    "--channels ignored"
    "Channels require claude.ai authentication"
    "Not logged in"
)
```

### CLI surface (additions)

| Flag | Behavior |
|---|---|
| `--reset` | rm -f today's counter file + cap-reached flag + not-logged-in flag; exit 0. Use after manual recovery (e.g. `/login`) to restore normal cooldown immediately rather than waiting for midnight. |
| `--status` | dump version, today's count vs CAP, both flag states, current effective cooldown, next-restart-allowed-at; exit 0. Debug aid. |
| `--show-config` (existing) | also prints the new env vars |

### Env vars (additions)

| Variable | Default | Description |
|---|---|---|
| `WATCHDOG_DAILY_RESTART_CAP` | `10` | Daily restart cap. `0` disables (legacy unbounded behavior). |
| `WATCHDOG_THROTTLED_COOLDOWN` | `3600` | Cooldown after cap (seconds). Subject to existing safety floor (`max(value, 2 × launchd_interval + 30s)`). |
| `WATCHDOG_ALERT_CMD` | *unset* | Shell expression invoked on alert. Receives `WATCHDOG_ALERT_TYPE` + `WATCHDOG_ALERT_MSG` env vars. Unset = log-only mode (alert still appears in `claude-watchdog.log` as `ALERT:` line). |

### Alert calling convention

```bash
WATCHDOG_ALERT_TYPE="$type" WATCHDOG_ALERT_MSG="$msg" sh -c "$WATCHDOG_ALERT_CMD"
```

`type` values for v0.1.6: `cap-reached` | `not-logged-in`. Future types via additive extension; readers should treat unknown types as `info`.

`msg` is human-readable, includes source identifier (e.g. hostname), symptom, and recovery action.

The convention deliberately does **not** pass the message as `$1` — passing via env avoids shell-escaping pitfalls when users compose `WATCHDOG_ALERT_CMD` strings that include `--data-urlencode` or similar argument-binding constructs.

## Downstream deployment guide

The alert hook is pluggable; this repo does not ship a default helper. Downstream installs configure their own. Below is the reference implementation used by the Mac mini dogfood target.

### Reference: dual-channel alert helper (macOS Notification + Telegram)

Save as `~/.claude/watchdog/alert.sh`, `chmod +x`, set:

```xml
<!-- in plist EnvironmentVariables -->
<key>WATCHDOG_ALERT_CMD</key>
<string>/Users/<you>/.claude/watchdog/alert.sh</string>
```

```bash
#!/bin/bash
# Dual-channel alert: macOS Notification + Telegram bot push
set +e

TYPE="${WATCHDOG_ALERT_TYPE:-info}"
MSG="${WATCHDOG_ALERT_MSG:-}"
[ -z "$MSG" ] && exit 0

# Channel 1: macOS notification (visible if anyone is at the screen)
/usr/bin/osascript -e \
    "display notification \"$MSG\" with title \"claude-watchdog [$TYPE]\"" \
    2>/dev/null

# Channel 2: Telegram (push to phone via existing bot infrastructure)
TG_ENV="$HOME/.claude/channels/telegram/.env"
if [ -f "$TG_ENV" ]; then
    . "$TG_ENV"
    if [ -n "${TELEGRAM_BOT_TOKEN:-}" ] && [ -n "${TELEGRAM_CHAT_ID:-}" ]; then
        /usr/bin/curl -fsS --max-time 10 \
            -X POST "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}/sendMessage" \
            --data-urlencode "chat_id=${TELEGRAM_CHAT_ID}" \
            --data-urlencode "text=[claude-watchdog ${TYPE}] ${MSG}" \
            >/dev/null
    fi
fi

exit 0
```

Notes:

- `--max-time 10` bounds the curl in case of network blip; well below launchd's 180s tick.
- `set +e` ensures alert failure never propagates to watchdog.
- `TELEGRAM_CHAT_ID` is read from the same `.env` as the bot token (downstream chooses how to populate).
- Standalone smoke test before wiring up: `WATCHDOG_ALERT_TYPE=test WATCHDOG_ALERT_MSG="hello" ~/.claude/watchdog/alert.sh`.

## Backward compatibility

Per CONTRIBUTING.md backward-compat contract:

**Public interfaces unchanged:**

- All existing env vars (`WATCHDOG_LOG_DIR`, `WATCHDOG_SESSION`, `WATCHDOG_COOLDOWN`, `WATCHDOG_PATH`, `WATCHDOG_CLAUDE_CMD`, `WATCHDOG_HEARTBEAT_FILE`, `WATCHDOG_HEARTBEAT_STALE_SECONDS`).
- Plist Label `com.openclaw.claude-watchdog`.
- Cooldown file format and path.
- Exit code 0 for daemon success.

**Public interfaces added (stabilized in v0.1.6):**

- `WATCHDOG_DAILY_RESTART_CAP`, `WATCHDOG_THROTTLED_COOLDOWN`, `WATCHDOG_ALERT_CMD` env vars.
- `--reset`, `--status` CLI flags.
- Alert protocol: `WATCHDOG_ALERT_TYPE` + `WATCHDOG_ALERT_MSG` env contract; type strings `cap-reached` | `not-logged-in`.
- State file paths and naming under `$LOG_DIR/`.

**Behavior changes:**

- v0.1.5 install upgraded to v0.1.6 gains `WATCHDOG_DAILY_RESTART_CAP=10` by default. Real-world impact: zero for healthy bots; pathological cases hit cap and write `ALERT:` to log (no alert delivery without `WATCHDOG_ALERT_CMD` configured). To preserve legacy unbounded behavior: set `WATCHDOG_DAILY_RESTART_CAP=0`.

### Source-guard at end of `claude-watchdog.sh`

```bash
if [ "${BASH_SOURCE[0]:-$0}" = "$0" ]; then
    main "$@"
fi
```

Tests `source claude-watchdog.sh` (env points to a temp `WATCHDOG_LOG_DIR`), then call functions directly.

### Integration harness

`PATH=$TEST_BIN:$PATH` prepends a temp dir with mock `tmux` and `pgrep`. Mock tmux reads canned responses from `MOCK_*` env vars, logs all calls to `MOCK_TMUX_LOG` so tests assert "kill-session was/wasn't called". Tests run the full script as a subprocess and observe state file changes, log lines, exit code.
