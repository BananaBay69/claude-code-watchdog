# claude-code-watchdog v0.1.7 — Silent-Loop Detection Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close [#15](https://github.com/BananaBay69/claude-code-watchdog/issues/15) — detect silent-loop (incoming channel messages arrive but no outbound `reply` tool call within window) and emit an `silent-loop` alert (no restart). Opt-in via `WATCHDOG_SILENT_LOOP_ENABLED=1`.

**Architecture:** Two-repo coordinated change with a file-based contract:
- **Plugin side** (`BananaBay69/claude-plugins/plugins/bananabay-watchdog`): new `outbound.sh` hook fires on `PostToolUse` matching `mcp__telegram__reply`; writes `1 <unix_ts>\n` atomically to `$WATCHDOG_OUTBOUND_FILE` (default `$HOME/.claude/watchdog/outbound`). Schema mirrors heartbeat for consistency.
- **Watchdog side** (`BananaBay69/claude-code-watchdog`): new Case D in `main()` after Case C; counts `← telegram · CHATID:` lines in `tmux capture-pane` (PoC), reads outbound file mtime, and if `incoming_count >= threshold && (now - outbound_mtime) > stale_seconds && pane clean` → emits `silent-loop` alert with state-based dedup (cleared when outbound advances).

**Why two repos one plan:** plugin writes the file, watchdog reads it. Coupled by contract but plan is staged so each phase ships independently — Phase A (plugin) can ship alone (no consumer); Phase B (watchdog) ships safely without plugin (silent-loop check no-ops when outbound file missing).

**Tech Stack:** bash 4+, tmux, Claude Code plugin hooks, shellcheck, GitHub Actions (macos-latest).

**Spec:** Issue [#15](https://github.com/BananaBay69/claude-code-watchdog/issues/15) (diagnosis comment + Step 0 verification comment).

**Working branches:**
- watchdog: `feat/v0.1.7-silent-loop-detection` (from `main` at v0.1.6)
- plugin: `feat/outbound-hook-v0.2.0` (from `main` at bananabay-watchdog v0.1.0)

**Out of scope** (file as separate issues):
- Discord side (`mcp__discord__reply`) — single Mr.Coconut process backs both, but incident was Telegram. Same contract trivially extends.
- Production-grade incoming source (Telegram `getUpdates` read-only). PoC pane-scrape ships in v0.1.7; production poller deferred to v0.1.8.
- Restart on silent-loop (kept alert-only — root causes are SKILL.md leaks, restart re-enters loop).
- Channels plugin not firing `UserPromptSubmit`/`Stop` for channel turns (upstream issue, file separately at `BananaBay69/claude-connect`).

## Cross-Repo Contract (locked here, not to drift during impl)

**Outbound state file** — written by plugin, read by watchdog.

| Field | Value |
|---|---|
| Path | `$WATCHDOG_OUTBOUND_FILE`, default `$HOME/.claude/watchdog/outbound` |
| Schema | `1 <unix_ts>\n` (single line, schema version 1, space-separated) |
| Write | atomic: `printf '1 %d\n' $(date +%s) > "$F.tmp" && mv -f "$F.tmp" "$F"` |
| Permissions | created with default umask; no special perms required |
| Trigger | every successful `PostToolUse` matching `mcp__telegram__reply` |
| Reader behavior | missing file = "disabled" (skip silent-loop check, log INFO once per tick); malformed = "stale" |

**`silent-loop` alert** — new alert type emitted by watchdog.

| Field | Value |
|---|---|
| Type string | `silent-loop` |
| Env to alert.sh | `WATCHDOG_ALERT_TYPE=silent-loop`, `WATCHDOG_ALERT_MSG="<incoming N> incoming msgs in pane, no outbound for <X>s — bot may be in silent loop. ssh into host and inspect pane."` |
| Dedup | state-based flag file `.watchdog-alert-sent-silent-loop`; cleared when outbound file advances past last-seen ts |
| Restart | **never** (alert only) |

```bash
# bananabay-watchdog — outbound hook.
# Fires on PostToolUse matching mcp__telegram__reply (and any other channel
# reply tool added by future configurations). Writes the v1 outbound format
# ("1 <unix_ts>\n") atomically to $WATCHDOG_OUTBOUND_FILE.

F="${WATCHDOG_OUTBOUND_FILE:-$HOME/.claude/watchdog/outbound}"

TMP="$F.tmp"
printf '1 %d\n' "$(date +%s)" > "$TMP" 2>/dev/null && mv -f "$TMP" "$F" 2>/dev/null

exit 0
```

- [ ] **B2.3 — Implement `outbound_state` (mirror `heartbeat_state`)**

Insert after `heartbeat_state()` in `claude-watchdog.sh` (around line 222):

```bash
# Echoes one of: disabled | fresh | stale
#
# Same v1 schema as heartbeat: "1 <unix_ts>". Read by Case D (silent-loop)
# to determine whether the bot has produced an outbound reply within the
# configured window.
#
# "disabled" reserved for "no signal available" — env unset or file missing —
# in which case Case D is skipped (cannot determine silent-loop without
# outbound signal).
outbound_state() {
    if [ -z "$OUTBOUND_FILE" ]; then
        echo "disabled"
        return
    fi
    if [ ! -f "$OUTBOUND_FILE" ]; then
        echo "disabled"
        return
    fi
    local schema ob_ts now age
    schema=""
    ob_ts=""
    # shellcheck disable=SC2162
    read schema ob_ts _rest < "$OUTBOUND_FILE" 2>/dev/null || true
    if [ "$schema" != "1" ]; then
        log "WARN: outbound unsupported schema '$schema' in $OUTBOUND_FILE — treating as stale"
        echo "stale"
        return
    fi
    if ! [[ "$ob_ts" =~ ^[0-9]+$ ]]; then
        log "WARN: outbound malformed timestamp '$ob_ts' in $OUTBOUND_FILE — treating as stale"
        echo "stale"
        return
    fi
    now=$(date +%s)
    age=$(( now - ob_ts ))
    if [ "$age" -gt "$SILENT_LOOP_OUTBOUND_STALE_SECONDS" ]; then
        echo "stale"
    else
        echo "fresh"
    fi
}
```

```bash
# Echoes "yes:<reason>" if silent-loop detected, "no:<reason>" otherwise.
#
# Args: $1 = incoming count (int)
#       $2 = outbound state (disabled|fresh|stale)
#
# Decision matrix (only fires when SILENT_LOOP_ENABLED=1):
#   incoming < threshold       -> no:below-threshold
#   outbound = disabled        -> no:no-outbound-signal (cannot determine)
#   outbound = fresh           -> no:outbound-fresh
#   outbound = stale           -> yes:incoming=N outbound-stale
detect_silent_loop() {
    local incoming="$1"
    local ob_state="$2"
    if [ "$SILENT_LOOP_ENABLED" -ne 1 ]; then
        echo "no:disabled"
        return
    fi
    if [ "$incoming" -lt "$SILENT_LOOP_INCOMING_THRESHOLD" ]; then
        echo "no:below-threshold(incoming=$incoming threshold=$SILENT_LOOP_INCOMING_THRESHOLD)"
        return
    fi
    if [ "$ob_state" = "disabled" ]; then
        echo "no:no-outbound-signal"
        return
    fi
    if [ "$ob_state" = "fresh" ]; then
        echo "no:outbound-fresh"
        return
    fi
    echo "yes:incoming=$incoming outbound-stale"
}
```
